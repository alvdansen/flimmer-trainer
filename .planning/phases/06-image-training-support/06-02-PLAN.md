<objective>
Add image_repeat config field and make the training sampler respect per-sample repeat counts, so users can control how often image samples appear in training batches relative to video samples.

Purpose: Images are 1-frame and train much faster than multi-frame videos. Without repeat balancing, a mixed dataset would undertrain on images. The image_repeat field lets users tune this balance (e.g., image_repeat=5 means each image appears 5x per epoch).

Output: New config field, expansion-time repeat multiplication, and sampler-level repeat expansion.
</objective>

  <action>
    The `BucketBatchSampler` currently groups indices by bucket key but never looks at `repeats`. The `repeats` field exists on `ExpandedSample` but is NOT on `CacheEntry` (the manifest model). The sampler receives a `CachedLatentDataset` which reads from the manifest.

    **Approach:** Add a `repeats` field to `CacheEntry` so it persists through the cache manifest. Then have `BucketBatchSampler` expand indices by repeat count.

    **In models.py:**

    1. Add `repeats: int = 1` field to `CacheEntry`:
    ```python
    repeats: int = Field(
        default=1,
        ge=1,
        description="Per-epoch repeat count for this sample (image_repeat * source repeats).",
    )
    ```
    This is backwards-compatible (default=1 means existing manifests without this field work fine).

    **In cache.py (minor):**

    2. In `build_cache_manifest()`, when creating CacheEntry objects, pass the `repeats` from the ExpandedSample:
    ```python
    entries.append(CacheEntry(
        ...,
        repeats=sample.repeats,
    ))
    ```
    Note: This file is NOT in files_modified for this plan because it's a one-line change. Add it if the executor needs to.

    **In dataset.py:**

    3. Add a `get_repeats(self, index: int) -> int` method to `CachedLatentDataset`:
    ```python
    def get_repeats(self, index: int) -> int:
        """Get the repeat count for a sample by index."""
        return self._entries[index].repeats
    ```

    4. Also expose a `repeats` property (list of repeat counts, parallel to `bucket_keys`):
    ```python
    @property
    def repeats(self) -> list[int]:
        """Repeat count for each sample index (for sampler construction)."""
        return [e.repeats for e in self._entries]
    ```

    5. Modify `BucketBatchSampler.__init__()` to expand indices by repeat count:
    ```python
    # Group indices by bucket key, expanding by repeats
    self._bucket_indices: dict[str, list[int]] = {}
    for idx, key in enumerate(dataset.bucket_keys):
        repeat_count = dataset.get_repeats(idx)
        self._bucket_indices.setdefault(key, []).extend([idx] * repeat_count)
    ```

    This means a sample with repeats=3 appears 3 times in its bucket's index list. When the sampler shuffles and batches, that sample gets picked ~3x per epoch. The DataLoader's `__getitem__` is called with the same index multiple times, which is correct -- it just reads the same cached tensors.

    **Also update cache.py** (add to files_modified if needed -- it's a one-line addition):
    In `build_cache_manifest()`, pass `repeats=sample.repeats` when creating each CacheEntry.

    - `__len__` reflects expanded batch count
    - A sample with repeats=3 appears 3 times in the iterated batches
    - Default repeats=1 produces identical behavior to before (regression test)
